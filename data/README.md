# Dataset — Task 1 Output

All CSVs in this directory were generated by the scripts in `../src/` against the
**public Polymarket Gamma API and data-api**. No private data, no API keys, fully
reproducible.

**Window of analysis:** Nov 1, 2025 – May 1, 2026 (per challenge requirements).

**Reproducibility.** Re-run from a clean checkout:

```bash
pip install -r ../requirements.txt
python ../src/discover_markets.py     # → markets.csv + markets_discovery_summary.json
python ../src/filter_markets.py       # → markets_filtered.csv + markets_filtered_summary.json
python ../src/backfill_trades.py      # → trades.csv + trades_backfill_summary.json
python ../src/enrich_wallets.py       # → wallets.csv + wallets_enrich_summary.json
```

Runtime is ~5–20 minutes depending on Polymarket API responsiveness.

## Files

### `markets.csv` — raw Gamma discovery (kept intact for audit)

Every event returned by the Gamma `/events` endpoint for any of the in-scope tag
IDs in `src/discover_markets.py:IN_SCOPE_TAG_IDS`. **Not filtered.** Includes
out-of-window events, low-volume markets, and bucket=`other` rows so the next
engineer can re-derive the filter step or apply different criteria.

| Column | Description |
|---|---|
| `event_id`, `event_slug`, `event_title` | Gamma event identity |
| `market_id`, `market_question`, `market_slug` | Individual binary market (one event can contain many markets for multi-outcome questions) |
| `condition_id` | The on-chain conditionId — primary key used to fetch trades from the data-api |
| `category`, `bucket` | Polymarket-assigned category + our classification (`reality_tv`, `awards`, `release_performance`, `other`) |
| `tag_id_source`, `tag_labels` | Which tag we found this event via, plus all tags Polymarket attaches |
| `start_date`, `end_date`, `closed_time` | Lifecycle timestamps (ISO 8601 UTC) |
| `active`, `closed`, `archived` | Lifecycle flags |
| `volume_usd`, `liquidity_usd` | Market-level lifetime volume and liquidity in USD |
| `outcomes` | Pipe-delimited outcome labels (e.g., `Yes|No`, or names for multi-outcome) |
| `resolved_outcome`, `resolved_price` | Winning outcome and its final price (1.0 = resolved YES) |
| `resolution_source` | URL Polymarket points to for adjudication |
| `market_maker_address` | LP / market-maker contract address |
| `ev_volume_usd_total` | Event-level total volume (sum across all child markets) |

### `markets_filtered.csv` — analysis set used by all downstream heuristics

Produced by `src/filter_markets.py`. Applies:
- `end_date ∈ [2025-11-01, 2026-05-01]`
- `closed == True` (resolved markets only)
- `volume_usd ≥ $50,000` (lifetime market volume floor)
- Hard exclude of sports/politics/crypto/fights/macro keywords
- Reclassification of `other` bucket using a richer pop-culture keyword set
- Drop anything still classified outside `{reality_tv, awards, release_performance}`

Same schema as `markets.csv` plus a (currently empty) `reveal_window_start`
column for Task 1 manual curation needed by heuristic A2.

### `trades.csv.gz` — all trades on filtered markets (gzipped, 28MB; ungzips to ~144MB)

Decompress with `gunzip -k data/trades.csv.gz` before running heuristics. The
raw `trades.csv` is gitignored to stay under GitHub's 100MB per-file limit.


One row per trade. Pulled from `data-api.polymarket.com/trades?market=<conditionId>`
with offset-based pagination (page size 500, cap of 10,000 per market).

| Column | Description |
|---|---|
| `trade_id` | Synthetic key: `tx_hash_timestamp_wallet`. Used for deduplication. |
| `timestamp_utc` | Trade time (ISO 8601 UTC) |
| `market_condition_id` | Links to `markets_filtered.csv.condition_id` |
| `event_id` | Links to `markets_filtered.csv.event_id` |
| `market_question`, `bucket` | Denormalized for convenience |
| `wallet_address` | Lowercased Polygon address of the proxy wallet (the trader) |
| `wallet_name`, `wallet_pseudonym` | Polymarket public profile name + pseudonym, if any |
| `side` | `BUY` or `SELL` |
| `outcome`, `outcome_index` | Which outcome was traded |
| `price` | Fill price (probability, 0–1) |
| `size_usd` | `price × shares` (notional USD) |
| `tx_hash` | Polygon transaction hash |

**Caveat — per-market trade cap.** Markets where the API returned ≥10,000 trades
are flagged in `trades_backfill_summary.json:per_market[].capped == true`. For
those markets coverage is partial; trade scoring should normalize accordingly.

**Caveat — empirical Polymarket data-api ceiling.** Several markets cap at
exactly 3,500 trades returned regardless of `offset` paging — likely a
server-side cap on `/trades` query depth. Visible on markets like
"Will Bruno Mars have the greatest number of monthly Spotify listeners this month"
(actual trade count > 3,500). For those markets the dataset under-samples
later-resolution trades. Mitigations: use the CLOB historical-prints endpoint
(`clob.polymarket.com/markets/.../trades?cursor=…`) for full coverage, or
treat affected markets as exploratory only in heuristic scoring.

### `wallets.csv` — per-wallet enrichment

One row per unique wallet appearing in `trades.csv`. Pulled from Polymarket
leaderboard endpoints.

| Column | Description |
|---|---|
| `wallet_address` | Primary key, lowercased |
| `pseudonym`, `name`, `bio`, `profile_image` | Polymarket profile metadata |
| `lifetime_profit_usd` | Lifetime realized P&L (USD) |
| `profit_30d_usd` | Trailing 30-day P&L (USD) |
| `lifetime_volume_usd` | Lifetime traded volume (USD) |
| `current_portfolio_value_usd` | Current open position value (USD) |
| `in_window_trade_count` | Count of trades this wallet has in `trades.csv` |

Used by heuristics:
- **B4** single-niche specialist: `in_window_volume / lifetime_volume_usd`
- **D8** Brier baseline: lifetime track record context
- **Anti-signals**: `lifetime_volume_usd` thresholding for market-maker
  identification; `name`/`pseudonym` for doxxed-sharp matching

## Summary JSON files

Every script writes a sidecar `*_summary.json` with reproducibility metadata
(generation timestamp, parameter values, counts by bucket/month, drop reasons,
per-market trade counts). These are also tracked in git so dataset provenance
travels with the repo.

| File | Produced by | Contents |
|---|---|---|
| `markets_discovery_summary.json` | `discover_markets.py` | Tag IDs queried, total events kept, bucket counts, per-tag event counts |
| `markets_filtered_summary.json` | `filter_markets.py` | Filter parameters, drop reasons, kept-by-bucket, kept-by-month, total volume |
| `trades_backfill_summary.json` | `backfill_trades.py` | Per-market trade counts, which markets hit the cap |
| `wallets_enrich_summary.json` | `enrich_wallets.py` | Endpoints used, success count |

## Known data caveats

1. **Multi-outcome events look noisy in `release_performance`.** A single event
   like "Top Spotify Artist 2025" generates ~30 child markets, one per
   candidate. They share liquidity and traders, so naive per-market
   aggregation will double-count. Downstream heuristics should be aware.
2. **`reveal_window_start` is unpopulated.** Heuristic A2 (pre-announcement
   timing cluster) needs this hand-attached. Suggested workflow: filter
   `markets_filtered.csv` to the top 20 by volume, attach reveal-window
   timestamps manually from public sources, score A2 only on the curated subset.
3. **Trade cap is per-market.** Markets above ~10K trades have only their first
   10K trades. The cap is documented per-market in the backfill summary; if
   the heuristics surface a wallet from a capped market, re-pull with a higher
   cap and re-score.
4. **Wallet enrichment uses public LB endpoints**, which lag the on-chain state
   by minutes and may report 0s for very fresh wallets. Acceptable for our
   purposes (we're looking at lifetime patterns, not point-in-time).
5. **Open events with `end_date` in window** were included in raw `markets.csv`
   but excluded from `markets_filtered.csv` (heuristics require resolved
   outcome). If you want to score open positions later, the raw file has them.
